# Prompt Rules Hub

Prompt Rules Hub — pet-проект про то, как frontend-команда может управлять правилами для AI-ассистентов так же аккуратно, как управляет кодом.

Вместо разрозненных заметок, сообщений в чатах и случайных промптов команда хранит правила в одном месте: с категориями, версиями, тегами и наследованием. Затем приложение собирает готовый Markdown-контекст, который можно передать ChatGPT, Codex, Cursor или внутреннему AI-инструменту.

## Проблема

AI-ассистент хорошо помогает только тогда, когда понимает правила команды. Но в реальности эти правила часто живут где попало:

- часть в README;
- часть в pull request comments;
- часть в личных заметках разработчиков;
- часть просто проговаривается на review.

Из-за этого один и тот же ассистент может писать компоненты в разном стиле, забывать про тесты, игнорировать дизайн-систему или давать review без нужного контекста.

## Решение

Prompt Rules Hub превращает правила для AI в структурированную библиотеку:

- каждое правило имеет категорию, версию, описание, теги и Markdown-контент;
- правила могут наследоваться друг от друга, чтобы не дублировать базовые инструкции;
- Context Builder подтягивает parent rules автоматически;
- итоговый контекст можно посмотреть, скопировать или скачать;
- данные хранятся локально в JSON, без базы данных.

Главная идея простая: инструкции для AI должны быть поддерживаемыми, версионированными и переиспользуемыми.

## Что внутри

- библиотека правил с поиском и фильтрами;
- страница правила с метаданными, родителями и Markdown;
- форма создания и редактирования правила;
- сборщик контекста с автоматическим подтягиванием parent rules;

- Express API с CRUD-эндпоинтами;

## Запуск

Установить зависимости:

```bash
npm install
```

Запустить приложение и API:

```bash
npm run dev
```

Адреса:

- интерфейс: `http://127.0.0.1:5174`
- API: `http://127.0.0.1:4000`

Если открыть `http://127.0.0.1:4000`, вы увидите служебную подсказку API. Основной интерфейс находится на `5174`, а данные доступны по маршрутам `/api/...`, например `http://127.0.0.1:4000/api/rules`.

Проверки:

```bash
npm run typecheck
npm run build
```

## Данные

Правила лежат в обычном JSON-файле:

```txt
data/rules.json
```

Это осознанный выбор для pet-проекта: легко показать seed data, объяснить storage-слой и обсудить, как заменить JSON на базу данных при росте продукта.

## Что показать на интервью

1. Откройте библиотеку правил и покажите категории: frontend, тестирование, review, дизайн-система и безопасность.
2. Откройте карточку правила и объясните модель: версия, теги, parent rules, Markdown-контент.
3. Создайте новое правило, например "Правила accessibility review", и привяжите его к базовому поведению ассистента.
4. Покажите, что frontend и backend используют одну Zod-схему.
5. В сборщике контекста выберите "Правила API компонентов" и покажите, как автоматически подтягиваются родительские правила.
6. Перейдите в экспорт и скачайте Markdown-контекст для AI-ассистента.
7. Откройте backend-код и объясните архитектуру: Express API, JSON storage, отдельный context composer.

## Что могут спросить

- Почему JSON storage? Так проще показать данные, storage-слой и путь миграции на базу.
- Как работает наследование? Выбранные правила подтягивают parent rules, чтобы не дублировать базовые инструкции.
- Зачем Zod? Одна схема валидирует данные и на frontend, и на backend.
- Почему Markdown и JSON? Markdown удобен для ChatGPT/Codex/Cursor, JSON — для автоматизаций.
- Как избежать циклов в parent rules? Проверять граф правил при сохранении и при compose.
- Как масштабировать проект? Добавить базу данных, авторизацию, роли, историю версий и review flow.
- Что важно для frontend-команды? Единый AI-контекст снижает случайность ответов ассистента и делает генерацию кода предсказуемее.
